二、预计算缓存（秒开）

生成脚本在部署时运行；前端优先读静态 JSON，服务端 API 同样命中内存/磁盘缓存，避免重复实时匹配。

缓存文件生成命令用途
/cache/sample-data.json npm run cache:showcase 首屏 55 公司 + 69 员工列表，秒开表格
/cache/full-match.json npm run cache:showcase AI 全量 55 家匹配结果；勾选公司客户端同步切片即时显示员工
/cache/manual-pool-back.json npm run cache:manual-pools 后道全量：41 家回访 × 48 人后道员工池
/cache/manual-pool-front.json npm run cache:manual-pools 前道全量：6 家首访 × 前道员工池
/cache/transit-routes.json 匹配时自动写入 高德公交/地铁路段磁盘缓存，串联行程复用

命中逻辑

  • AI 页：tryInstantFullMatchCache() — 勾选公司后从已预载的 full-match.json 切片，不等待 session 即可显示员工
  • 手动页：tryManualPoolCacheHit() — 后道/前道全选或子集勾选，若 ⊆ 预计算池则客户端秒开；否则调 API，服务端 tryGetManualPoolDispatch 命中
  • 服务端：无 employeePoolIds 时走 tryGetFullMatchDispatch；有员工池时走 tryGetManualPoolDispatch（支持 exact + subset 切片）
  • 版本：dataVersion 与 integrated-cache.ts 一致，变更后须重新生成缓存

三、数据层（匹配前）

函数：buildIntegratedData（src/data/integrated-data.ts）

  1. 读取 data/ 目录 Excel：首访 / 项目 / 回访 / 员工表
  2. 追加 10 家演示公司 + 5 名演示员工（姓名带 演示- 前缀，如「演示-范金山」）
  3. 追加 37 名补位员工（补位-*，补下午2、外埠园区、出发地就近容量缺口）
  4. 应用 员工补丁（gap-fill-employees.ts）：修正出发地、扩展时段（如金山常用 15 人补下午2）
  5. 校验全员姓名唯一：Excel + 演示 + 补位不能同名（assertUniqueEmployeeNames）

整合结果：55 家公司、69 名员工（27 Excel + 5 演示 + 37 补位）。首屏可读缓存 /cache/sample-data.json。

四、单条「公司 → 员工」合规规则

核心函数：matchCustomerToEmployee（src/services/match-rules.ts）

对每一对客户，在该员工当前已分配的其他单上下文下逐条判定：

#规则类型说明
1城市匹配硬约束 customer.cityId === employee.cityId
2区域匹配硬约束 从客户拜访地址与员工出发地推断省份（inferProvinceFromAddress），须一致；禁止跨省派单（如上海员工不能接山东济南客户）
3职责匹配硬约束 首访→前道 FRONT；项目→PROJECT；回访→后道 BACK
4时段匹配硬约束 员工 orderCapacity 含该时段；且该时段尚未被该员工其他单占用
5指定人硬约束 有指定人必须派给指定人；指定人在候选池则不能派给别人
6放弃人硬约束 不能派给放弃人字段中的员工
7园区匹配硬约束 员工出发地能否地理覆盖客户所属园区名（canDepartureServePark）；外埠园区（山东济南、江苏徐州等）须员工出发地在对应省
8下午捆绑硬约束 后道：同园区下午1+下午2 须同员工（在优化器层原子处理，成功后写入规则明细）
9Plus 匹配软约束 AI / 手动模式默认不要求（requirePlus: false）；Excel 全量模式可要求

合规判定

AI / 手动：7 条核心规则（城市、区域、职责、时段、指定人、放弃人、园区；不含 Plus）全部通过 → eligible = true。下午捆绑在优化器层单独校验。

得分（选优用，非硬失败）

基础分 100 + 指定人匹配 +50 + 园区覆盖 +10 + Plus 适配加分 最多 +20 − 该员工已接单数 × 5

五、一人多单与容量模型

优化器维护 assignedPerEmp：Map<员工 ID, 已接客户列表>。

  • 同一员工最多：上午 1 单 + 下午1 1 单 + 下午2 1 单（共 3 单，须不同时段）
  • 时段冲突是硬约束，不是「一人只能接 1 单」
  • 手动页可展示「需求时段 vs 员工池容量」预警（容量不足时提示，不硬拦截请求）

六、后道下午捆绑

文件：afternoon-park-pairs.ts + pairing-optimizer.ts

业务规则：后道回访，同一园区内，下午1 与 下午2 必须派给同一名员工（该员工下午跑 2 单）。

处理步骤

  1. 按 parkName 分组，拆成下午1 列表 / 下午2 列表
  2. 贪心配对（按指定人优先、Plus 优先级排序）
    • 两单指定人不同 → 不强行绑定
    • 数量不等（如 9 下午1 vs 8 下午2）→ 多出的单单独匹配
  3. 对每个绑定对，找能同时合规接下午1+下午2 的员工，选 combinedPairScore 最高者
  4. 原子写入 assignedPerEmp（一次占满两个时段）
  5. 失败 → 两条都标未匹配，原因「下午捆绑未满足…」

配对得分（2026-06 更新：通勤最短优先）

新规则（默认）：在全部合规员工中，串联通勤最短者优先；同通勤再比 match.score。

通勤时间来源：必须使用 commuteMode: transit（高德公交/地铁 API + 磁盘缓存）。

90 分现实上限：选优时优先 ≤90 分候选；仅当全员超标才允许 >90（指定人单例外）。60 分为 UI 软提示，不导致匹配失败。

路线合理性：高德 API 返回多条方案时，过滤绕远线路（机场线/磁浮短途、路径距离超直线倍数等）；短途优先公交，极短距离才用步行。

下午捆绑：选下午1+下午2 串联总通勤最短者；同总通勤再比 score 之和。

客户/捆绑对处理顺序：指定人 → Plus 优先级 → 预约时间（与旧规则相同）。

旧规则（preferShortestCommute: false）：

combinedPairScore = combinedScore(下午1) + combinedScore(下午2)

combinedScore = match.score − 本段串联通勤 × 0.2（Plus/园区分可压过通勤差）

七、全局优化算法 findCapacitatedMatching

文件：src/services/pairing-optimizer.ts

① 处理 lockedPairings（AI 页已锁定 / 手动指定的配对，先占位） ② 后道下午捆绑：成对 → 原子分配给同一员工（两单总通勤最短） ③ 对其余客户：按 sortCustomersForDispatch 排序后逐单贪心 ④ 每单：在 eligible 员工中选串联通勤最短 → 同通勤比 match.score ⑤ 未匹配分析：最近候选、失败规则、时段冲突员工 ⑥ 输出 pairings + employeeSchedules（上午→下午1→下午2 + 段间路线）

客户排序 sortCustomersForDispatch

  1. 有指定人的客户优先
  2. Plus 低→高（Plus0 → Plus1 → PlusN）
  3. 预约时间早→晚

通勤

  • 串联路程：一人多单时，本段通勤 = 上一站地址 → 本单地址（首单仍为出发地 → 客户）；拜访顺序为 上午 → 下午1 → 下午2
  • 匹配选优（默认）：合规池中串联通勤最短 → 同通勤比 match.score；优先 ≤90 分候选
  • 通勤来源：高德公交/地铁（commuteMode: transit）+ transit-routes.json 磁盘缓存；配置 GAODE_API_KEY 后自动启用
  • 无高德 key 时降级本地矩阵估算；生产环境应配置 key 以保证跨省/绕远判断准确
  • 超过 60 分钟不导致匹配失败，UI 显示软提示；>90 分在选优阶段尽量排除
  • 员工行程 routeSegments 与单条 commuteMinutes 均与串联逻辑一致

八、AI 匹配页流程（match.html）

8.1 数据加载

  1. 并行预载 sample-data.json + full-match.json（秒开列表与全量缓存）
  2. 后台 bootstrapIntegratedData 创建 session，之后可调 API

8.2 选公司与自动预览

  1. 用户勾选公司 → selectedCompanies
  2. 优先 tryInstantFullMatchCache()：从内存中的 full-match 缓存同步切片，右侧立即显示员工（无需等 API）
  3. 缓存未覆盖时，防抖 400ms 调用 POST /api/dispatch/select（员工池=全员，commuteMode: transit）
  4. 结果写入 assignmentMap，渲染看板 + 员工行程 Tab

8.3 增量匹配（锁定 + 只重算未锁定）

用户手动改派某行后，该行 manual: true。下次预览传：

{ "sessionId": "...", "customerIds": [全部仍选中的公司], "lockedPairings": [{ "customerId": 101, "employeeId": 7 }, ...], "matchOnlyCustomerIds": [103, 104, ...], "commuteMode": "transit" }
  • lockedPairings：已成功匹配且仍被选中的公司（含未改动的自动匹配行）
  • matchOnlyCustomerIds：选中公司 − 已锁定公司
  • 若全部锁定、无待重算 → 不调 API，仅本地展示

8.4 手动改派（单条校验）

下拉换员工 → POST /api/dispatch/validate-pair，传入 existingPairings（其他已分配）。

  • 同一套 matchCustomerToEmployee 规则
  • 前端额外检查：该员工该时段是否已被其他公司占用
  • 合规 → 写入 assignmentMap，触发对其余单的增量重算

8.5 一键全量匹配

优先读预生成 /cache/full-match.json 客户端秒开；缓存缺失时再调 API（fullMatch: true）匹配 55 家。服务端同样命中 full-match 内存缓存。

8.6 历史

结果存浏览器 localStorage（dispatch-ai-history），不写服务器数据库。

九、手动派单页流程（manual-match.html）

9.1 操作

  1. 首屏读 sample-data.json 秒开公司与员工表格
  2. 可点「后道全选」/「前道全选」一键选中对应全量公司与员工池
  3. 或手动表格勾选公司 + 可选员工池（不选则用全员）
  4. 点击「确认匹配」→ 优先客户端 tryManualPoolCacheHit 秒开，否则 POST /api/dispatch/select
{ "sessionId": "...", "customerIds": [选中的公司 ID], "employeePoolIds": [7, 8, 9, ...], // 可选 "commuteMode": "transit" }

9.2 前置校验

  • 员工池不能为空（若指定了 pool）
  • 员工池姓名不能重复（前端 + 后端双重校验）
  • 全选员工时，同名只保留一条
  • 公司不能重复选择

9.3 结果

  • 成功 / 失败列表
  • employeeSchedules：多单员工全天行程
  • 历史：dispatch-manual-history（localStorage）

十二、Excel 上传旧引擎（另路）

POST /api/dispatch → DispatchEngine（dispatch-engine.ts）

  • 不经过 pairing-optimizer
  • 前道+项目 / 后道分开派单
  • 支持牵手单（备注含「牵手单」的客户整组同员工）
  • 无下午捆绑、无AI 页锁定机制

线上 AI 匹配与手动派单均不走此引擎。

十三、完整示例：金山 32 家回访 + 15 人池

输入

  • 园区：加盟-金山资本现代产业园
  • 上午 15 家 + 下午1 九家 + 下午2 八家 = 32 家后道回访
  • 员工池 15 人（李路路、傅丽、姚洁…，补丁后含下午2 容量）
Step 1 下午捆绑预处理 → 8 对（同园区下午1+下午2） → 1 家单独下午1（9 vs 8 余单：得韬会展） Step 2 8 对原子分配 → 各找 1 名员工同时合规接两单 → 例：傅丽 → 筱胜（下午1）+ 鸣旭（下午2） Step 3 其余 24 单贪心匹配 → 指定人客户优先 → 例：姚洁 → 惠铎（下午1，指定）+ 宝屋（下午2） Step 4 构建 employeeSchedules → 傅丽：上午 1 + 下午1 + 下午2 = 3 单 → 盛雅琴：上午 1 + 单独下午1 = 2 单 输出：32/32 成功（容量充足时）

十四、未匹配原因类型

原因含义
无合规员工（城市/区域/职责/时段/指定人/放弃人/园区） 7 条核心规则中有不满足项；常见：区域不匹配（跨省）、园区不匹配（出发地无法覆盖园区）
下午捆绑未满足 同园区下午对找不到能同时接两单的员工
该时段可用员工已满 有合规员工，但该时段已被其他单占满
串联通勤超标（>90 分） 合规候选全员单程/串联 >90 分钟且无指定人豁免时，可能无法派单
员工池容量不足 手动页预警：需求时段 > 池内该时段可用人数
员工姓名重复 手动选池含同名员工，API 拒绝
