一、正式环境功能清单

正式环境指服务器上运行的派单 Web 应用，数据以 Excel 主数据 为来源，匹配结果可导出，不写回 Excel 源文件。

页面 / 模块路径用途
正式 Excel 上传派单 / · index.html 上传员工表 + 首访/项目/回访 Excel，一次性全量派单，下载结果
正式 选择园区派单 /select.html 上传后按园区筛选公司，批量派单
正式 AI 匹配 /match.html 加载整合数据，勾选公司 AI 自动匹配；支持本地通勤（默认）加速
正式 手动派单 /manual-match.html 勾选公司 + 锁定部分配对 + 增量匹配；可调员工池、查看行程
正式 Excel 对比 /compare-dispatch.html 系统派单结果与历史 Excel 指派对比
测试 匹配测试报告 /test-match.html 合成/全国模拟场景 + 即时试算；不替代正式派单

正式环境核心能力

  • 从 Excel 导入园区、员工、待派单客户（主数据 INPUT）
  • 按 7 条规则筛选合规员工，支持一人多单（同一员工不同时段各 1 单）
  • 在合规员工中按得分 + 通勤距离选最优（容量约束匹配）
  • 输出派单结果 + 员工当日行程表（派单后 OUTPUT）
  • 通勤估算：本地矩阵（默认）或 DeepSeek API（可选，余额不足时自动降级本地）

三、数据流

3.1 主数据（派单前 INPUT）

实体Excel 来源关键字段
园区 InvestmentPark园区数据.xlsx（可选，也可从客户单补全）园区名称、城市、地址
员工 Employee派单员工表 (1).xls姓名、出发地、负责园区、职责、时段容量、Plus 能力
客户 Customer首访/项目/回访数据.xlsx公司名、拜访地址、所属园区、类型、时段、指定人/放弃人

原则：园区与员工是独立主数据；员工的「负责园区」仅用于规则匹配，表示服务范围，不是组织归属。

3.2 整合数据管线（AI 匹配 / 手动派单默认加载）

Excel 五表 importAllData() │ ├─► 原始客户 + 员工 + 园区 │ ├─► + 演示数据 showcase（带 [演示] 标签） │ ├─► + 补位员工 gap-fill（带 [补位] 标签，保证 55 家全量可匹配） │ └─► applyEmployeePatches() 修正出发地/时段 │ ▼ getIntegratedData() 内存缓存（Excel 变更自动失效） │ ▼ 前端 sessionId ──► POST /api/dispatch/select

3.3 派单后 OUTPUT

  • 派单结果：每条 = 1 客户 → 1 员工，含通勤、得分、规则明细
  • 员工行程表：按员工汇总当日单量（上午/下午）、路线段、总通勤
  • 导出：POST /api/export 生成 Excel 下载

3.4 端到端流程（AI 匹配页）

用户打开 match.html → GET /api/sample-data（或 bootstrap） → 展示 55+ 家公司、51+ 名员工 → 用户勾选公司 → POST /api/dispatch/select { customerIds, commuteMode: 'local' } → pairing-optimizer 容量约束匹配 → 返回 pairings + employeeSchedules + unmatched → 前端展示成功条 / 未匹配原因

四、匹配逻辑（7 条规则）

函数：matchCustomerToEmployee(customer, employee, availableNames, assignedCustomers)

正式派单与测试试算共用同一实现（src/services/match-rules.ts）。

#规则类型说明
1城市匹配硬约束 customer.cityId === employee.cityId，必须同城
2职责匹配硬约束 首访→前道 FRONT；项目→项目 PROJECT；回访→后道 BACK
3时段匹配硬约束 员工 orderCapacity 含该时段；且该时段尚未被该员工其他单占用（一人多单核心）
4指定人硬约束 有指定人时必须派给指定人；指定人在候选池中则不能派给别人
5放弃人硬约束 不能派给放弃人字段中的员工
6园区匹配硬约束 员工出发地能否地理覆盖客户所属园区名（canDepartureServePark）
例：园区「浦东-张江」→ 出发地需含浦东/张江等地域词；「加盟-金山…」→ 需含上海或金山
7下午捆绑硬约束 后道回访：同一园区内，下午1 与 下午2 必须派给同一名员工（下午跑 2 单）
指定人冲突的两单不强行配对；数量不等时（如 9 下午1 vs 8 下午2）剩余单可单独匹配
8Plus 匹配软约束 选择派单 / 测试模式默认不要求 Plus；Excel 全量派单模式可要求

合规判定

全部硬约束通过 → eligible = true。得分用于在多个合规员工中选优（指定人加分、园区覆盖加分、已接单数减分）。

通勤（优化用，非硬失败）

合规员工间优先选通勤更短者；超过 60 分为软提示，指定人等情况仍可匹配成功。

五、派单优化与一人多单

函数：findCapacitatedMatching（pairing-optimizer.ts）

  1. 客户按指定人优先、Plus 优先级排序
  2. 后道下午捆绑：同园区下午1+下午2 先成对，再原子分配给同一员工（afternoon-park-pairs.ts）
  3. 对其余客户（上午、未成对的下午单），在当前已分配上下文下找所有 eligible 员工
  4. 维护 assignedPerEmp：每位员工已接客户列表
  5. 同一员工：上午 / 下午1 / 下午2 各最多 1 单（时段匹配第 3 条）
  6. 在合规员工中选 score - 通勤×0.2 最高者
  7. 无法匹配时分析原因：无合规员工 / 时段已满 / 下午捆绑无法满足

输出 employeeSchedules：按 上午→下午1→下午2 排序，计算段间路线与总通勤。

六、举例说明

例 1 · 金山亭林 · 首访 + 上午 → 0 人（规则正确）

输入：园区「加盟-金山资本现代产业园」或「金山-亭林」；地址「上海金山区亭林路10号」；首访；上午

现象：亭林镇有多名员工（姚焕、李路路等），通勤约 21 分，但 合规 0 人

原因：

  • 这些员工职责为 后道，首访需要 前道 → 职责不匹配
  • 补位员工「补位-韩金山」等仅有 下午2 容量 → 上午时段不匹配
  • 园区匹配、城市匹配通常 通过

若改为回访 + 上午：可匹配 20+ 名后道员工，推荐亭林出发的姚焕等。

例 2 · 杨浦复旦 · 首访 + 上午 → 需前道员工

输入：园区「杨浦-复旦」；地址「上海市杨浦区邯郸路220号」；首访；上午

现象：宋樑、盛雅琴（杨浦出发）不能接首访 — 他们是后道

解决：补位员工「补位-杨浦前」（前道+项目，全时段，邯郸路出发）可合规匹配；或客户类型改为回访由后道接单。

例 3 · 一人多单 · 同一员工一天 3 单

场景：张全能（浦东张江出发，前道+后道+项目，全时段）

时段客户类型结果
上午A 公司首访✓ 张全能
下午1C 公司项目✓ 张全能
下午2E 公司首访✓ 张全能

若再派一个上午客户给张全能 → 时段已满，引擎改派李备岗或标记未匹配。

例 4 · 指定人

客户指定「黄健」，候选池中有黄健 → 只能派给黄健；派给他人则指定人规则失败。

指定人不在候选池 → 可派给其他合规员工，并在得分上优先非指定场景。

例 5 · 园区匹配（地理覆盖，非名称相等）

客户园区「浦东-张江」，员工出发地「上海市浦东新区张江路100号」→ 通过（地域词覆盖）

客户园区「浦东-张江」，员工出发地「上海市闵行区莘庄镇」→ 失败（出发地无法覆盖园区）

推荐园区命名：区域-地标，如「浦东-张江」「金山-亭林」。

例 6 · 正式全量 55 家匹配

数据源：Excel 45 家 + 演示 10 家；员工含补位后约 52 人。

AI 匹配页勾选全部 → POST /api/dispatch/select → 典型结果 55/55 全匹配，支持员工一天多单，均合规。

七、正式 API 一览

方法路径说明
POST/api/dispatch上传 Excel 一次性派单
POST/api/parse-data解析上传，返回 session + 元数据
GET/api/sample-data加载整合数据（正式默认数据集）
GET/api/bootstrap轻量 session，数据来自内存缓存

POST/api/export导出派单结果 Excel
POST/api/test-match-lookup测试 单地址试算，不落库
GET/api/test-match-report测试 批量场景报告
